# SLED_SMEN.md

Единый журнал следов рабочих чатов.

Один чат = одна запись.
Одна запись = один номер.
Заголовок записи = короткое имя чата, которое можно вручную перенести в название треда.

## Как вести журнал

1. Первое рабочее действие нового чата: открыть этот файл, взять следующий свободный номер, придумать короткое имя из первого сообщения пользователя и сразу создать запись.
2. Короткое имя делается максимально коротким. Обычно 2-4 слова. Формат заголовка: `### 001 Журнал следов`.
3. Сразу после заголовка агент заводит черновой след в той же записи, даже если основная работа ещё не началась.
4. Только после этого агент читает другие файлы, собирает контекст и делает основную задачу.
5. В том же чате нельзя заводить второй номер. Запись всегда переписывается как общий итог всего чата на текущий момент.
6. Чужие записи не редактировать. Закрытые чаты не трогать.
7. Если в чате всплыли несколько близких решений, держать одну основную суть, а остальные перечислять кратко в той же записи.
8. Если работа явно ушла в другую рабочую задачу, агент мягко подсказывает, что для новой дороги лучше открыть новый чат.
9. Перед долгим расследованием технической или продуктовой проблемы агент быстро просматривает блоки `Проблемы` ниже, чтобы не повторять уже пройденный путь.
10. Если долгой проблемы раньше не было, агент после решения кратко дописывает её в свой блок `Проблемы`: что произошло и как вышли.
11. Формулировки держать короткими и рабочими. Без витрины, без самоописания, без лишней истории.

## Формат записи

- `### 001 Журнал следов`
- `**Что уложено** ...`
- `**Статус слоя** Мутно / Идти можно / Внятно`
- `**Что стало внятнее** ...`
- `**Что ещё мутно** ...`
- `**Передача смены** ...`
- `**Проблемы**`
- `- Нет.` или короткие пункты вида `что было -> как решили`

Для самого первого чернового следа в новом чате достаточно коротких формулировок:

- `**Что уложено** След открыт. Идёт работа.`
- `**Статус слоя** Идти можно`
- `**Что стало внятнее** Зафиксированы номер и короткое имя чата.`
- `**Что ещё мутно** Основная работа ещё впереди.`
- `**Передача смены** После содержательной работы обновить эту же запись.`
- `**Проблемы**`
- `- Нет.`

**Статус слоя** Идти можно

**Что стало внятнее** Появилось одно место, где хранится итог по каждому рабочему чату; агентам задано, когда брать номер, как сразу фиксировать короткое имя вида `001 Журнал следов`, как обновлять только свою запись и когда смотреть прошлые проблемы перед долгим расследованием.

**Что ещё мутно** Не проверено на нескольких последовательных чатах вживую; если позже всплывёт неудобство с длиной коротких имён или со слишком ранним созданием черновых следов, правило ещё можно дожать без смены общей логики.

**Передача смены** Дальше важно не вернуть в `AGENTS.md` дублирование следа в финальном ответе; следующий агент должен первым делом открыть `SLED_SMEN.md`, завести заголовок вроде `002 Короткое имя`, а этот чат при продолжении обновляет только `001 Журнал следов`.
 
**Проблемы**
- Нет.

**Что уложено** Рабочее состояние переведено на `main`; в `AGENTS.md` и `README.md` зафиксировано правило одной ветки, коммитов только по явной команде пользователя и нумерации коммитов по номеру смены; добавлен `presentations-frontend/vercel.json`, который отключает автодеплой веток кроме `main`; расширен `.gitignore`, чтобы локальные профили, логи и тестовые хвосты не попадали в общий коммит; production-сборка проходит через `npm run verify`.
