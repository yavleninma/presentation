# SLED_SMEN.md

Единый журнал следов рабочих чатов.

Один чат = одна запись.
Одна запись = один номер.
Заголовок записи = короткое имя чата, которое можно вручную перенести в название треда.

## Как вести журнал

1. Первое рабочее действие нового чата: открыть этот файл, взять следующий свободный номер, придумать короткое имя из первого сообщения пользователя и сразу создать запись.
2. Короткое имя делается максимально коротким. Обычно 2-4 слова. Формат заголовка: `### 001 Журнал следов`.
3. Сразу после заголовка агент заводит черновой след в той же записи, даже если основная работа ещё не началась.
4. Только после этого агент читает другие файлы, собирает контекст и делает основную задачу.
5. В том же чате нельзя заводить второй номер. Запись всегда переписывается как общий итог всего чата на текущий момент.
6. Чужие записи не редактировать. Закрытые чаты не трогать.
7. Если в чате всплыли несколько близких решений, держать одну основную суть, а остальные перечислять кратко в той же записи.
8. Если работа явно ушла в другую рабочую задачу, агент мягко подсказывает, что для новой дороги лучше открыть новый чат.
9. Перед долгим расследованием технической или продуктовой проблемы агент быстро просматривает блоки `Проблемы` ниже, чтобы не повторять уже пройденный путь.
10. Если долгой проблемы раньше не было, агент после решения кратко дописывает её в свой блок `Проблемы`: что произошло и как вышли.
11. Формулировки держать короткими и рабочими. Без витрины, без самоописания, без лишней истории.

## Формат записи

- `### 001 Журнал следов`
- `**Что уложено** ...`
- `**Статус слоя** Мутно / Идти можно / Внятно`
- `**Что стало внятнее** ...`
- `**Что ещё мутно** ...`
- `**Передача смены** ...`
- `**Проблемы**`
- `- Нет.` или короткие пункты вида `что было -> как решили`

Для самого первого чернового следа в новом чате достаточно коротких формулировок:

- `**Что уложено** След открыт. Идёт работа.`
- `**Статус слоя** Идти можно`
- `**Что стало внятнее** Зафиксированы номер и короткое имя чата.`
- `**Что ещё мутно** Основная работа ещё впереди.`
- `**Передача смены** После содержательной работы обновить эту же запись.`
- `**Проблемы**`
- `- Нет.`

**Что уложено** Система слайдов переведена с одинаковых 3-блочных карточек на 6 разных визуальных компонентов по образцу демо-презентации из Figma v2: cover (крупный заголовок + мета), metrics (карточки с крупными числами), steps (нумерованные шаги 01/02/03 с коннекторами), checklist (строки с галочками), personas (карточки-персоны с ролью/задачей/тегом), features (иконка + заголовок + описание). В `presentation-types.ts` расширен `CanvasLayoutId` (6 новых значений) и `SlideBlock` (опциональные `metric`, `stepNumber`, `tagline`). В `demo-generator.ts` каждая slide function привязана к своему layout, `buildBlocks` разведён на 6 отдельных builder-функций, добавлен `extractMetricValues` для вытаскивания чисел из фактов. В `slide-canvas.tsx` вместо единого `blocks.map()` — switch по layout с отдельным JSX для каждого компонента; `SlideThumbnail` показывает мини-превью по типу layout. В `globals.css` — полный набор стилей для всех 6 компонентов с поддержкой шаблонов (strict/cards/briefing) и цветовых тем (slate/indigo/teal/sand). Исправлен баг `splitAudience`, который разрезал «Руководитель направления» по каждой букве «и». `npm run verify` зелёный, все 6 слайдов проверены визуально через cursor-ide-browser на двух промптах (бедный и богатый).

**Статус слоя** Внятно

**Что стало внятнее** Каждый из 6 слайдов теперь визуально отличается от остальных: cover — чистый титульник, metrics — крупные числа, steps — шаговая последовательность, checklist — список с галочками, personas — карточки-роли, features — фичи с иконками. При богатом промпте с цифрами (12%, 200мс, 40%) система вытаскивает числа в метрики автоматически. При бедном промпте показывает placeholder'ы.

**Что ещё мутно** Метки description в метриках иногда обрезаются неудобно (показывают «Конверсия выросла на» вместо полного текста). Personas при одном адресате (не перечислении) падают в fallback с placeholder'ами. Десктопная проверка >1200px не проводилась.

**Передача смены** Не возвращать одинаковые 3-блочные карточки. При доработке метрик можно улучшить `extractMetricValues` — парсить единицы измерения точнее. Personas полезно доработать: если адресат один, вместо fallback показывать другой layout (например, тот же checklist или features).

**Проблемы**
- `splitAudience` разрезал «Руководитель направления» по каждой кириллической «и» → исправлен regex на `\s+и\s+` (только союз «и» как отдельное слово).
