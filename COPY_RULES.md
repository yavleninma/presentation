# COPY_RULES.md

## Зачем этот файл

Этот файл — единственный источник правды по текстам в интерфейсе.
Агент обязан сверяться с ним перед тем, как писать любой видимый пользователю текст.

---

## Главное правило

Текст в интерфейсе — про задачу пользователя.
Не про продукт. Не про устройство продукта. Не про философию продукта.

Если текст можно показать человеку вне контекста продукта и он всё равно полезен — текст хороший.
Если текст имеет смысл только как описание продукта — он мутный и должен быть убран.

---

## Запрещённый внутренний словарь

Эти слова и фразы **никогда не должны появляться в интерфейсе**:

- "первый слой", "слой", "следующий слой"
- "канонический сценарий", "канонический"
- "артефакт"
- "мутно", "внятно"
- "граница слоя", "осознанная граница"
- "несущий каркас"
- "экран N", "маршрут"
- "Codex", "агент", "модель"
- "дизайн-спайн", "design spine"
- "проходимость"

Это внутренний язык проекта. Пользователь его не знает и знать не должен.

## Три антипаттерна текста

### 1. Текст-комментатор

Текст объясняет, что делает экран, вместо того чтобы экран просто делал.

| Плохо | Хорошо |
|---|---|
| "Система уже поняла, что именно нужно показать и кому." | "Мы правильно поняли?" или убрать совсем |
| "Хребет будущего deck уже собран и не требует выбора темы или формата." | "План презентации" |
| "Черновик собирается по шагам, а не прячется за спиннером ради спиннера." | "Собираю черновик" |
| "На этом слое продукт делает только одно: принимает рабочий запрос..." | Убрать. Поле и кнопка уже говорят это. |
| "Пользователь уже видит, где именно система находится и какой артефакт появится следующим." | Убрать. Сам экран это и показывает. |

**Тест:** если текст описывает дизайн-решение ("мы показываем X, а не Y"), это комментатор. Убрать.

### 2. Текст-оправдание

Продукт объясняет пользователю, чего он не умеет. Если кнопки нет — функции нет, и объяснять нечего.

| Плохо | Хорошо |
|---|---|
| "Нельзя редактировать, экспортировать или дозаполнять фактуру." | Убрать. |
| "Что ещё мутно: здесь пока нет редактирования..." | Убрать. |
| "Только просмотр" | Убрать. |
| "Это осознанная граница первого слоя." | Убрать. |

**Тест:** если текст начинается со слов "нельзя", "пока нет", "здесь ещё не...", это оправдание. Убрать.

### 3. Текст-витрина

Продукт продаёт себя пользователю, который уже внутри продукта.

| Плохо | Хорошо |
|---|---|
| "Промпт вместо формы" | Убрать. |
| "Рабочая презентация без фальшивого экспорта" | Убрать. |
| "Один вход / один путь / один результат" | Убрать. |
| "Первый слайд для руководителя должен читать суть без экскурсии по процессу." | Убрать. |

**Тест:** если текст объясняет, почему продукт хорош или чем он отличается, это витрина. Убрать.

## Лимиты текстовых элементов на экран

Каждый экран имеет бюджет. Бюджет считается по видимым текстовым блокам (заголовок, параграф, карточка, chip, label).

| Экран | Элементы | Состав |
|---|---|---|
| Старт | 4–5 | Заголовок + поле + кнопка + примеры промптов |
| Понимание | 3 | Цитата запроса + карточки данных + кнопка |
| План | 2 | Список из 6 пунктов + кнопка |
| Сборка | 2 | Список шагов с галочками + мини-превью |
| Редактор | 2 | Лента миниатюр + холст со слайдом |

Если на экране больше элементов, чем в бюджете, убирай лишние, начиная с тех, которые описывают продукт.

## Правило контента слайдов

Текст на слайде — это текст **реальной презентации**, которую увидит руководитель.
Не комментарий о том, каким должен быть хороший слайд.
Не описание дизайн-решения.
Не пересказ философии продукта.

| Плохо (заголовок слайда) | Хорошо (заголовок слайда) |
|---|---|
| "Фактура встроена в артефакт, а не вынесена в отдельную валидацию" | "Ключевые метрики Q1 2026" |
| "Основная работа разложена по трём понятным блокам" | "Что сделали за Q1" |
| "Риски вынесены в отдельный разговор, а не спрятаны в мелкий текст" | "Риски и блокеры" |

| Плохо (тело слайда) | Хорошо (тело слайда) |
|---|---|
| "Черновик показывает не только движение вперёд, но и то, где команда удержала качество." | "Мигрировали 18 сервисов. Снизили MTTR на 30%. Перевели CI на новый пайплайн." |
| "Слайд оставляет место для управленческого контекста." | Конкретный текст из запроса пользователя. |
| "Черновик уже держит структуру разговора." | Убрать. Слайд сам это демонстрирует. |

**Правило:** если текст слайда можно показать руководителю как часть настоящей презентации — он хороший. Если текст имеет смысл только как описание продукта — он мутный.

## Правило step rail / навигации

Step rail (если есть) показывает **только названия шагов** максимально коротко.
Не нужно описание каждого шага. Не нужны мета-комментарии.

| Плохо | Хорошо |
|---|---|
| "Старт: Принять сырой рабочий запрос." | "Запрос" |
| "Понимание: Показать, что система правильно поняла задачу." | "Понимание" |
| "Черновик: Открыть собранные слайды без лишней поверхности." | "Черновик" |

---

## Section pills / labels

Section pills не должны содержать номера экранов или мета-описания.

| Плохо | Хорошо |
|---|---|
| "Экран 2. Понимание задачи" | Убрать или "Понимание" |
| "Экран 3. План презентации" | Убрать или "План" |
| "Экран 5. Черновик презентации" | Убрать |
| "Канонический сценарий" | Убрать |
| "Граница этого слоя" | Убрать |

---

## Итоговый чеклист перед отдачей UI-текста

Перед каждым экраном прогони:

1. Говорит ли этот текст о задаче пользователя или о продукте? Если о продукте — убрать.
2. Есть ли в тексте слова из запрещённого словаря? Если да — убрать.
3. Можно ли убрать этот текст без потери смысла экрана? Если да — убрать.
4. Звучит ли заголовок как действие или как описание дизайн-решения? Если второе — переписать.
5. Сколько текстовых элементов на экране? Если больше бюджета — убрать лишние.
6. Текст слайда: покажет ли его пользователь руководителю? Если нет — переписать.
