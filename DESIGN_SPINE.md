# DESIGN_SPINE.md

## Актуализация UI-опоры

HTML-референс `references/redesign-2026-04/preview.html` из репозитория снят. Визуальная и продуктовая опора для текущего прототипа — **Figma v3 — Brand Kit** и раздел **«v3 — Brand Kit»** в `DESIGN_DECISIONS.md` (см. также `AGENTS.md`).

Пока этот документ не приведён к каркасу v3 целиком, для маршрута и поведения действуют такие правила:

- при конфликте с v3 по **пикселям, токенам и хрому** приоритет у **Figma v3 и `DESIGN_DECISIONS.md`**;
- видимые состояния маршрута по смыслу: тихий вход, короткое уточнение, отдельная сборка, черновик, редактор с закрытым drawer, редактор с открытым drawer;
- `fit-pass` остаётся скрытым служебным проходом после генерации, а не пользовательским экраном;
- внутренние пометки агента вроде «Экран N» из старых материалов не копируются в UI.

## Как читать этот документ

Документ содержит два типа текста:
- **Спецификации** — маршрут, роли экранов, принципы, состав слайдов. Это опора для кода.
- **Мотивировки** — почему именно так. Контекст для понимания, но они не должны попадать в интерфейс ни в каком виде.

Агент использует спецификации как основу для реализации. Мотивировки помогают понять намерение, но их текст **не копируется** в UI.

## Зачем это нужно
*(мотивировка)*

Проекту не хватает **дизайнового позвоночника**: маршрута, ролей экранов, визуальной грамматики и понятного первого результата.

## Что делаем сейчас
*(спецификация)*

Строим **design spine v0**:

1. один маршрут пользователя;
2. один язык экранов;
3. один UI-kit v0;
4. один набор канонических слайдов;
5. один внятный способ передавать это Codex.

## Главный маршрут
*(спецификация)*

**Старт → короткое уточнение → сборка → черновик → редактор**

Скрытый служебный проход **fit-pass** выполняется после генерации и не является пользовательским экраном.

Полная схема маршрута — в `SCREEN_ATLAS.md`.

## Канонический сценарий
*(спецификация)*

**Показ продукта Внятно** (первый тестовый сквозной сценарий прототипа).

## Главный результат
*(спецификация)*

За один рабочий сеанс пользователь получает собранный черновик из 6 слайдов:

1. Обложка
2. Проблема
3. Три шага
4. Результат
5. Для кого
6. CTA / Следующий шаг

## Роли экранов
*(спецификация)*

У каждого экрана одна роль. Детальные схемы — в `SCREEN_ATLAS.md`.

### Старт — тихий вход
Роль: принять рабочий запрос одним полем.

### Уточнение
Роль: коротко собрать то, чего не хватает для сильного первого черновика. Уточнение не обязательно по глубине: можно ответить, а можно сразу запускать сборку.

### Сборка
Роль: честно показать, что система сейчас собирает черновик, а не переносить пользователя на пустой экран.

### Черновик
Роль: показать текстовый каркас слайдов, удержать смысловую опору из уточнения и дать уточнить содержание перед переходом в редактор.

### Редактор (drawer закрыт)
Роль: дать работать с конечными слайдами.

### Редактор (drawer открыт)
Роль: AI-доработка активного слайда и заметки к нему.

## Что обязательно держать в голове
*(мотивировка — не копировать в UI)*

### Внятно
- в каждый момент ясно, где пользователь;
- в каждый момент ясно, что делает система;
- в каждый момент ясно, что будет дальше;
- основной объект уже виден как можно раньше.

### Мутно
- пользователь видит только форму и проверку;
- система не показывает путь;
- вместо артефакта показывает счётчики, пробелы, диагностические блоки;
- экран больше похож на админку или чеклист, чем на продукт.

## Принципы дизайна v1
*(спецификация — воплощать в поведении, не в тексте UI)*

1. **Чат сначала, поля потом**
   Поля появляются только как уточнение внутри процесса, а не как стартовый барьер.

2. **Сначала артефакт, потом диагностика**
   Человек должен сначала увидеть собранный результат, а не список того, чего не хватает.

3. **Один визуальный мир**
   Нельзя, чтобы старт был тёплым брифом, а редактор внезапно становился другим продуктом.

4. **Один спокойный хром**
   Старт, уточнение, сборка, черновик и редактор не должны ощущаться как разные продукты.
   Важен один светлый рабочий контур без резких визуальных переломов.

5. **Не спорить с презентацией**
   Самый важный объект на экране — слайд.
   Не карточка статуса, не панель, не диагностика.

6. **Иконка = роль, не украшение**
   Иконки нужны для навигации по смыслу: метрики, риски, команда, фокус.

7. **Точечные placeholders вместо пустых валидаций**
   Недостающие данные живут внутри слайда как мягкая подсказка.

## Что нельзя делать
*(спецификация)*
- большой стартовый бриф на 6–10 полей;
- счётчики вроде 0/8 как главный смысл экрана;
- показывать пробелы раньше, чем появился артефакт;
- разные визуальные языки на соседних этапах;
- пытаться решить маршрут одними текстами без схем экранов.

*(мотивировка)*
Codex нельзя отдавать только словами «сделай как Kimi, но по-своему».
Ему нужно давать:

- маршрут экранов;
- схемы экранов;
- токены и правила UI;
- список компонентов;
- список запретных паттернов;
- 3–5 референсных скринов;
- один канонический сценарий.

Тогда он перестаёт гадать.
