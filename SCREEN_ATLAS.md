# SCREEN_ATLAS.md

Схематичная карта экранов прототипа в `presentations-frontend`.
Нужна, чтобы маршрут и роли экранов были видны без чтения кода.

Опора по визуалу — **Figma v3 — Brand Kit** и **`DESIGN_DECISIONS.md`** § v3.
Опора по маршруту — **этот документ** и фактическое поведение кода прототипа.

Правила чтения:

- ASCII-схемы показывают структуру и порядок, а не пиксели.
- Тексты в схемах каноничны. Если схема говорит `Открыть редактор →`, в коде должно быть именно это.
- Внутренние пометки вроде «Экран N» в UI не попадают.

---

## Маршрут прототипа

```
[Старт]
    ↓ ввод запроса
[Уточнение]                      ← screen = "clarify"
    ↓ «Собрать черновик →»
[Сборка]                         ← screen = "building"
    ↓
[Черновик: сводка + сетка + чат] ← screen = "draft"
    ↓ «Открыть редактор →»
[Редактор]                       ← screen = "editor"
    ↓ Sparkles
[Редактор + панель]
```

Скрытый служебный проход **fit-pass** выполняется после генерации и не является пользовательским экраном.

## Экран 1. Старт
**Роль:** принять рабочий запрос одним полем и не заставлять выбирать лишнее.
**Код:** `PrototypeScreen = "start"`

```
┌──────────────────────────────────────────────────────────────┐
│  ◼ Внятно                                                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│              О чём презентация?                              │
│                                                              │
│  ┌──────────────────────────────────────────────────── [↑] ─┐│
│  │ Опишите, что нужно показать, кому и какой следующий...  ││
│  └──────────────────────────────────────────────────────────┘│
│                                                              │
│                     [Показ продукта]                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

Что важно:
- один рабочий chip, без фальшивых вариантов;
- нет ранней диагностики и лишнего текста;
- ошибка первого запроса остаётся здесь же, без перехода на пустой экран.

## Экран 2. Уточнение
**Роль:** коротко собрать то, чего не хватает для сильного первого черновика.
**Код:** `PrototypeScreen = "clarify"`

```
┌──────────────────────────────────────────────────────────────┐
│  ◼ Внятно                                                    │
├──────────────────────────────────────────────────────────────┤
│  ┌─── chat-card ───────────────────────────────────────────┐ │
│  │  Уточним главное перед черновиком                      │ │
│  │  Можно ответить коротко или сразу идти дальше.         │ │
│  │  ─────────────────────────────────────────────         │ │
│  │  [Вы] Нужно показать сервис Внятно...                  │ │
│  │  [AI] Тема и аудитория уже понятны.                    │ │
│  │       Можно собирать черновик уже сейчас.              │ │
│  │       [Покажи пользу для команды]                      │ │
│  │       [Добавь следующий шаг]                           │ │
│  │                                                        │ │
│  │                    [Собрать черновик →]                │ │
│  │  ─────────────────────────────────────────────         │ │
│  │  Добавьте короткое уточнение...                  [↑]   │ │
│  └────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

Что важно:
- первый ответ помощника открывает путь дальше, а не ставит барьер;
- быстрые ответы помогают добрать смысл без длинного брифа;
- можно не продолжать чат и сразу запускать сборку.

## Экран 3. Сборка
**Роль:** показать, что система реально собирает черновик.
**Код:** `PrototypeScreen = "building"`

```
┌──────────────────────────────────────────────────────────────┐
│  ◼ Внятно                                                    │
├──────────────────────────────────────────────────────────────┤
│  ┌─── build-status ────────────────────────────────────────┐ │
│  │  Собираем черновик                                      │ │
│  │  Короткая сводка по собранной опоре                     │ │
│  │  ─────────────────────────────────────────────          │ │
│  │  01 Собираем тему и адресата                    идёт    │ │
│  │  02 Выделяем факты и следующий шаг             ждёт     │ │
│  │  03 Раскладываем историю по 6 слайдам          ждёт     │ │
│  │  04 Проверяем первый рабочий проход            ждёт     │ │
│  │                                                        │ │
│  │  Слайды готовы: 3 из 6                                 │ │
│  └────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
```

Что важно:
- это отдельное состояние, а не пустой draft;
- видно, что делает система и какой шаг идёт сейчас.

## Экран 4. Черновик
**Роль:** показать собранный текст по слайдам, удержать смысловую сводку и дать поправить содержание перед редактором.
**Код:** `PrototypeScreen = "draft"`

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Что держим в фокусе: краткая сводка            │  Помощник              │
│  Аудитория · следующий шаг · факты              │  Уточняйте, меняйте... │
├─────────────────────────────────────────────────┼────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │  [AI] Черновик готов.  │
│  │ Обложка  │  │ Проблема │  │ Три шага │       │  [Вы] Правка...        │
│  │ title    │  │ title    │  │ title    │       │  [AI] Обновил...       │
│  └──────────┘  └──────────┘  └──────────┘       │                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │  Напишите пожелание... │
│  │ Результат│  │ Для кого │  │ След.шаг │       │             [Отправить]│
│  └──────────┘  └──────────┘  └──────────┘       │                        │
├─────────────────────────────────────────────────┴────────────────────────┤
│                         [ Открыть редактор → ]                           │
└──────────────────────────────────────────────────────────────────────────┘
```

Что важно:
- сверху остаётся та же смысловая опора, что собрана в уточнении;
- inline-правки карточек должны попасть в editor без потери;
- главный следующий шаг назван честно: это переход в редактор.

**Роль:** работать с конечными слайдами и локально усиливать активный слайд.
**Код:** `PrototypeScreen = "editor"`

```
┌──────────────────────────────────────────────────────────────────────────┐

├────────────┬─────────────────────────────────────────────────────────────┤
│  [01] ▓▓▓ │                                                             │
│  [02] ▓▓▓ │                       СЛАЙД 16:9                             │
│  [03] ▓▓▓ │                                                             │
│  [04] ▓▓▓ │                                                             │
│  [05] ▓▓▓ │                                                             │
│  [06] ▓▓▓ │                                                             │
├────────────┴─────────────────────────────────────────────────────────────┤
│                    [←]  Слайд 1 из 6  [→]                                │
└──────────────────────────────────────────────────────────────────────────┘
```

При открытом drawer справа появляется панель `Настройки слайда` с AI-действиями и заметками.

---

## Что проверять на каждом экране

1. Ясно ли, где находится пользователь?
2. Ясно ли, что делает система?
3. Ясно ли, какой следующий шаг сейчас главный?
4. Не потерялась ли смысловая опора между этапами?
5. Нет ли декоративного шума вместо полезного действия?
