# SCREEN_ATLAS.md

## Актуализация 2026-04

Для текущего редизайна главным экранным референсом временно считается `references/redesign-2026-04/preview.html`.

Пока этот atlas не переписан под новый каркас:

- подписи `Экран 1`, `Экран 2`, `Экран 3`, `Экран 4` и `Скрытый слой` в HTML нужны агенту для извлечения маршрута и не должны попадать в UI;
- пользовательскими состояниями считаются только тихий вход, короткий мини-чат и редактор в двух вариантах: drawer closed / drawer open;
- скрытый `fit-pass` остаётся внутренним проходом после генерации и не выносится в отдельный экран;
- всё, что ниже конфликтует с новым HTML-референсом, требует последующего приведения к нему, а не обратного переопределения референса.

Ниже не финальные макеты, а **схематичные экраны**.
Они нужны, чтобы Codex и человек видели маршрут и распределение ролей.

**Тексты в вайрфреймах — каноничны.** Это не черновик для замены. Если в вайрфрейме написано "Собираю черновик", в коде должно быть "Собираю черновик", а не "Черновик собирается по шагам, а не прячется за спиннером ради спиннера". Агент может улучшить формулировку, но не раздувать и не добавлять мета-комментарии. Все правила текстов — в `COPY_RULES.md`.

## Экран 1. Старт
**Роль:** принять сырой запрос и запустить сборку.

```text
┌──────────────────────────────────────────────────────────────────────┐
│ Лого / Внятно                                         История / ... │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  Соберите квартальный статус команды                                 │
│  без длинного брифа и пустого листа                                  │
│                                                                      │
│  [ большое поле: "Опишите что произошло за квартал..."          ]   │
│                                                                      │
│  Сценарии: [Квартальный статус] [Talent review] [Инициатива]         │
│                                                                      │
│  [ Собрать черновик ]                                                │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

Не должно быть:
- 8 полей;
- счётчиков пробелов;
- большой валидации;
- signal cards, объясняющих устройство продукта ("Один вход", "Один путь", "Один результат");
- объяснительных параграфов о том, что делает продукт;
- composer notes типа "Этот слой заканчивается на...";
- section pills с внутренним словарём ("Канонический сценарий").

## Экран 2. Понимание задачи
**Роль:** показать, что система правильно поняла запрос.

```text
┌──────────────────────────────────────────────────────────────────────┐
│  Ваш запрос                                                          │
│  "Собери квартальный статус backend platform team за Q1..."       │
│                                                                      │
│  Понял задачу так:                                                   │
│  • Период: Q1 2026                                                   │
│  • Команда: Backend Platform                                         │
│  • Аудитория: руководитель направления                               │
│  • Цель: показать прогресс, риски и следующий шаг                    │
│  • Тон: строгий, рабочий                                             │
│                                                                      │
│                         [ Собрать план ]                             │
└──────────────────────────────────────────────────────────────────────┘
```

Здесь всё коротко.
Не надо превращать в длинный анализ запроса.

Не должно быть:
- заголовков типа "Система уже поняла, что именно нужно показать и кому";
- мета-описаний "Здесь нет длинного анализа. Есть только те опоры...";
- section pills с номерами экранов ("Экран 2. Понимание задачи").

---

## Экран 3. План презентации
**Роль:** показать структуру будущего объекта.

```text
┌──────────────────────────────────────────────────────────────────────┐
│  План презентации                                                    │
│                                                                      │
│  01 Обложка                                                          │
│  02 Главный вывод и запрос к руководителю                            │
│  03 Ключевые метрики                                                 │
│  04 Что сделали за период                                            │
│  05 Риски и блокеры                                                  │
│  06 Фокус следующего периода                                         │
│                                                                      │
│                [ Подтвердить и собрать черновик ]                    │
└──────────────────────────────────────────────────────────────────────┘
```

Это должен быть уже не туман, а понятный хребет будущего deck.

Не должно быть:
- honesty aside с описанием ограничений ("Что здесь уже можно, а что ещё нет");
- текстов-оправданий ("Нельзя редактировать, экспортировать...");
- мета-заголовков типа "Хребет будущего deck уже собран и не требует выбора темы".

## Экран 4. Сборка
**Роль:** показать движение, а не спиннер ради спиннера.

```text
┌──────────────────────────────────────────────────────────────────────┐
│  Собираю черновик                                                    │
│                                                                      │
│  [✓] Понял задачу                                                    │
│  [✓] Собрал план                                                     │
│  [~] Подбираю визуальный ритм                                        │
│  [ ] Генерирую слайды                                                │
│                                                                      │
│  Мини-preview справа:                                                │
│  ┌──────────────────────────────┐                                    │
│  │  cover draft                 │                                    │
│  └──────────────────────────────┘                                    │
└──────────────────────────────────────────────────────────────────────┘
```

Тут нужно ощущение реальной работы системы.

Не должно быть:
- мета-текстов "а не спиннер ради спиннера";
- описательных параграфов о том, что видит пользователь;
- section pills с номерами экранов.

## Экран 5. Редактор
**Роль:** дать работать уже со слайдами.

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Back   Название документа                             Save   Preview Export │
├──────────────┬───────────────────────────────────────────────────────────────┤
│ [01]         │                                                               │
│ [02]         │                      СЛАЙД / ХОЛСТ                             │
│ [03]         │                                                               │
│ [04]         │                                                               │
│ [05]         │                                                               │
│ [06]         │                                                               │
│              │                                                               │
├──────────────┴───────────────────────────────────────────────────────────────┤
│ [ Напиши: "сделай короче", "добавь риск", "перепиши под CTO" ]    [→]    │
└──────────────────────────────────────────────────────────────────────────────┘
```

Очень важно:
- слева миниатюры;
- центр доминирует;
- чат доработки компактный;
- нет тяжёлой формы справа.

Не должно быть:
- мета-футера с описанием состояния продукта ("Что уже уложено", "Что ещё мутно");
- бейджей "Только просмотр";
- section pills с номерами экранов;
- текстов-оправданий о том, чего продукт не умеет.

## Экран 6. Недостающая фактура внутри артефакта
**Роль:** добрать недостающее без развала результата.

```text
┌──────────────────────────────────────────────────────────────────────┐
│  Слайд: Главный вывод                                                │
│                                                                      │
│  "Команда удержала стабильность и снизила долг,                     │
│   но следующий квартал упрётся в приоритет найма QA"                │
│                                                                      │
│  • Uptime стабилен                                                   │
│  • Техдолг снижен                                                    │
│  • [ Нужна 1 цифра по time-to-release ]                              │
│                                                                      │
│  Запрос к руководителю:                                              │
│  [ Уточни, какое решение нужно: найм / бюджет / приоритет ]          │
└──────────────────────────────────────────────────────────────────────┘
```

Диагностика встраивается в сам объект.
Не живёт отдельно как главный экран.

---

## Экран 7. Экспорт / финал
**Роль:** завершить работу.

```text
┌──────────────────────────────────────────────────────────────────────┐
│  Черновик готов                                                      │
│                                                                      │
│  6 слайдов                                                           │
│  2 места отмечены как требующие фактуры                              │
│                                                                      │
│  [ Открыть редактор ]   [ Экспорт PPTX ]   [ Экспорт PDF ]           │
└──────────────────────────────────────────────────────────────────────┘
```

Не надо делать отдельный сложный экран ради экспорта.

---

## Что проверять на каждом экране
1. Ясно ли, где находится пользователь?
2. Ясно ли, что делает система?
3. Ясно ли, что будет следующим шагом?
4. Показываем ли мы объект как можно раньше?
5. Нет ли здесь мутного диагностического шума?
